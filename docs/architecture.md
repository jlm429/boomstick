# Architecture

Boomstick is a browser-first, single-player arena combat prototype.

| Area                                            | Ownership                                |
| ----------------------------------------------- | ---------------------------------------- |
| Application screens, menu, HUD, pause dialog    | React                                    |
| Authoritative application phase and run state   | Zustand                                  |
| Scene graph, camera, lighting, visual animation | React Three Fiber / Three.js             |
| Gravity, colliders, player rigid body           | Rapier                                   |
| Movement, encounter, and combat rules           | Rendering-independent TypeScript utility |

`App` switches between React screens using one phase machine: `main-menu`, `about`, `arena-entry`, `playing`, `paused`, and `training-complete`. The native Canvas is passed directly to `App`; Enter Arena, Resume, and Restart request pointer lock from their click handlers. The browser's `pointerlockchange` event alone confirms active play, and lock loss pauses the run. Store transitions preserve the invariant that `playing` always has pointer lock and every other phase does not.

`GameViewport` keeps arena and combat presentation outside Rapier's asynchronous boundary. The rendering-independent layouts in `src/game/arena.ts` and `src/game/targets.ts` supply arena geometry, repeated light-fixture placement, and fixed targets; arena blocks always have matching Rapier colliders, while target colliders exist only through the training countdown and light fixtures use forgiving rendered hit volumes without physics colliders. `CombatScene` owns transient weapon, target health, encounter progress, and broken-light state. It delegates ammunition and reload rules to `src/game/shooting.ts`, encounter transitions to `src/game/encounter.ts`, and reports weapon and encounter presentation state to the React HUD. `RunScene` owns the encounter-dependent physics composition, including the target colliders and single zombie, while `src/game/zombie.ts` owns the zombie health and movement tuning. Zombie corpse removal advances the encounter to completion, which pauses the scene, releases pointer lock, hides the HUD, and shows the React completion overlay.

`CombatScene` accepts fire and reload input only during pointer-locked play and raycasts each fired pellet against rendered scene geometry so the nearest collision controls hit feedback. A live target hit uses the raycast hit point in target-local space and its distance from the shot origin to calculate damage through the pure rules in `src/game/targets.ts`; depleted targets remain visible but stop participating in shot raycasts. Surface, live-target, and live-zombie hits select wall impact audio, while a light hit within the inclusive 18-unit destruction limit extinguishes that fixture and selects its one-time destruction audio. The zombie uses a dynamic Rapier body while alive, reads the player camera position, probes fixed collider geometry for direct or alternate steering, and switches explicitly among idle, chase, attack, hit, and dead behavior. Zombie hits spawn short-lived impact-point blood particles, while each bite animation spawns one smaller burst near the camera when it reaches the lunge impact. Death stops movement, collision, and bite audio immediately, then removes the corpse and owned runtime resources after the death animation and hold. Remounting `RunScene` for a new `runId` resets the weapon, targets, lights, countdown, zombie, physics bodies, animation state, particles, and audio. Returning to the main menu unmounts the Canvas and performs the same cleanup. A Rapier startup delay therefore cannot remove the sky, lighting, visible level, targets, or weapon. The fixed-step physics world is paused outside play. `Player` installs keyboard and mouse listeners once per mount, clears held input on pause, blur, and visibility loss, and uses refs for transient controls. The render loop guards transforms, resets escaped bodies, clamps unsafe frame transitions, and follows the rigid body at eye height.

The reproduced black-screen report was a rendered but unreadable scene rather than a missing Canvas or a player escaping the level. Browser inspection found a nonzero native Canvas, a healthy WebGL2 context, advancing frames, initialized Rapier, and a finite player at the configured spawn. The old near-black background, fog, and low-contrast geometry looked black once the entry panel disappeared, while any pointer-lock loss correctly disabled movement. The fix uses an opaque renderer, a bright background, readable standard materials and lighting, direct pointer-lock ownership, and visible geometry that no longer depends on physics startup.

This boundary keeps gameplay additions testable and prevents UI work from coupling to physics presentation.
